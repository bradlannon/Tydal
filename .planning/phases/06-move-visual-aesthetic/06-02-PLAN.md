<task type="auto">
  <name>Task 1: RGB pad coloring system</name>
  <files>public/ui/pad-grid.js, public/styles.css</files>
  <action>
Add a pad coloring system to pad-grid.js that assigns background colors based on each pad's musical role. This applies to the note-zone pads (bottom 4x8).

**Color scheme (Ableton Move default track 1 = orange-amber):**
- Root note pads: `#e87a20` (Move's orange) with subtle glow `0 0 8px rgba(232, 122, 32, 0.4)` — these are the brightest
- In-scale pads (non-root): `#2a2a2a` (dim warm gray) — visible but subdued
- Out-of-scale / chromatic pads: `#0d0d0d` (nearly black, barely distinguishable from background) — essentially unlit
- Active (pressed) pad: `#ffffff` (bright white) — overrides all other colors

**Implementation in pad-grid.js:**

1. Add a `_applyPadColors()` function that iterates all `.note-cell` elements in the grid:
   ```
   For each note-cell:
     Extract the note name from cell.dataset.note (e.g. "C4")
     Extract pitch class (e.g. "C")
     If scaleLock is active:
       If pitch class matches scaleLock.key -> root color
       If pitch class is in the scale notes -> in-scale color
       Else -> out-of-scale color
     Else (chromatic mode, no scale lock):
       All pads get in-scale color (#2a2a2a) — no root distinction without a key center
   Set cell.style.backgroundColor and cell.style.boxShadow accordingly
   ```

2. Call `_applyPadColors()` at the end of `_createGrid()` after the note zone is built (use requestAnimationFrame to ensure DOM is ready).

3. Also call `_applyPadColors()` from `_rebuildGrid()` so color updates when scale/octave changes.

4. Update the `.note-cell.active` CSS class: set background-color to `#fff`, box-shadow `0 0 12px rgba(255, 255, 255, 0.5)`. This must use `!important` or higher specificity to override the inline backgroundColor set by the coloring system. Best approach: in `setPadActive()`, directly set `cell.style.backgroundColor` to `#fff` when active and restore the role color when inactive. Store the role color in a `data-role-color` attribute.

5. Update `setPadActive(note, isActive)`:
   ```
   if (isActive):
     cell.style.backgroundColor = '#fff'
     cell.style.boxShadow = '0 0 12px rgba(255, 255, 255, 0.5)'
   else:
     cell.style.backgroundColor = cell.dataset.roleColor || '#2a2a2a'
     cell.style.boxShadow = cell.dataset.roleGlow || 'none'
   ```

6. Export a `TRACK_COLOR` constant set to `#e87a20` so Phase 8 (multi-track) can swap it per track later.

**CSS changes in styles.css:**

7. Update `.note-cell` base styles:
   - Remove any background color (will be set inline by JS)
   - border: `none`
   - border-radius: `4px`
   - Remove hover effect entirely (`@media (hover: hover) .note-cell:hover` — delete it)

8. Update grid proportions for Move layout (MLAY-03):
   - `.note-zone` gap: `3px` (was `4px` — tighter like hardware)
   - `.note-zone` grid rows: `repeat(4, 1fr)` stays, but add `min-height` to give pads a wider-than-tall aspect ratio. Set `aspect-ratio` on `.note-cell` to roughly `1.2 / 1` (Move pads are slightly wider than tall). If aspect-ratio conflicts with grid fill, use a different approach: set `.note-zone { grid-auto-rows: 1fr; }` and let the grid cell height be determined by available space, then constrain `.note-cell` to `max-height: calc(100% - 2px)` with width filling the column.
   - Actually the simplest approach for Move proportions: the existing `flex: 3` on `.note-zone` already makes pads larger than steps. Just ensure the gap is tight (3px) and border-radius is small (3px). The 4x8 grid at `flex: 3` vs step zone `flex: 2` already gives a good Move-like proportion.

9. Remove `.cell-note` display (should already be `display: none` from Plan 01, verify).
